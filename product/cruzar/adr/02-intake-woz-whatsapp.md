# ADR-02 — Intake is WOZ copy-paste over WhatsApp

**Status:** Accepted
**Date:** 2026-04-15
**Owner:** Enrique (decision), Miura (operator)

## Context

MVP 0 captures 40+ intake questions per student to drive the readiness classifier and per-category plan generator. Candidate delivery channels:

1. In-product web form or conversational UI.
2. WhatsApp via Meta Cloud API (webhook, templates, business verification).
3. WhatsApp via Miura operating a personal/cohort account, pasting prompts generated by Claude Code.

WhatsApp is U0's native channel — Peruvian students respond to WhatsApp immediately, to email slowly, to web forms not at all. In-product web forms lose to WhatsApp on engagement. Meta Cloud API is simple but still adds: business verification, phone number provisioning, template approval (1–4h wait), webhook plumbing, session-state keyed by phone number, outbound message costs. ~4–6h of work that produces the same questions in the same order.

At a 5-user cap (U0 Fase 1), the automation premium of Cloud API doesn't pay off. A human operator pasting takes ~20 min per student spread across 1–2 days.

## Decision

Intake is Wizard-of-Oz. Miura drives it from Claude Code sessions:

- `/cruzar intake <id> --generate` — CC reads prior batches + answers from DB, generates the next batch of 10 adaptive questions, outputs them formatted for WhatsApp paste. Persists prompt text on `intake_batches`.
- Miura copies → pastes into WhatsApp → sends student.
- Student replies async (hours to 24h).
- `/cruzar intake <id> --record` — Miura pastes the reply; CC parses into structured Q→A on `intake_batch_answers`; preserves `raw_reply` verbatim for audit.
- Repeat 4 times → 40+ questions captured → `/cruzar intake <id> --finalize` transitions the student into assessment.

No Meta Cloud API integration in MVP 0. No webhook. No templates. No business verification. Miura's personal WhatsApp (or a dedicated cohort number) is the transport.

## Alternatives considered

- **Meta WhatsApp Cloud API.** Correct for Phase 1 continuation when cohort exceeds Miura's paste bandwidth or when engagement data (read receipts, automated nudges) becomes valuable. ~4–6h of work. Too much surface area for 24h relative to the savings at 5 users.
- **In-product web intake.** Would keep everything in `apps/web` and skip WhatsApp, but LatAm student engagement on cold web forms is structurally worse than WhatsApp. Traction numbers would drop.
- **Email intake.** Slower than WhatsApp, higher drop-off, harder to iterate conversationally.

## Consequences

Positive:
- Zero WhatsApp infra. `students.whatsapp` is the only column needed.
- Premium feel: questions are adaptive (CC sees prior answers, crafts the next 10 accordingly) and feel human because Miura is literally sending them.
- Audit trail: `raw_reply` captured verbatim on every batch.
- Miura's total cost: ~20 min per student over 2–3 days (student-paced).

Negative:
- Doesn't scale past 10 simultaneous students without Miura's paste time becoming the bottleneck.
- If Miura mistypes or pastes the wrong student's questions, the error is human. Mitigated by `/cruzar intake --generate` always echoing the student name and last 2 Q/A pairs at the top of the paste payload.
- No automatic nudge on stalled students — Miura follows up manually. Acceptable at 5 users.

## Sunset criteria

Revisit when:

- Cohort exceeds 10 simultaneous students, or
- Stall rate (students not replying within 48h) exceeds 30% and automated nudges become valuable, or
- Universities start asking for intake analytics that WOZ can't produce.

On sunset, default migration is Meta WhatsApp Cloud API with the same orchestrator behind a webhook. Shared intake logic — change the transport, not the brain.
